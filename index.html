Features

  • WebGL counts the percentage of sessions that have successfully created at least one WebGL context, among the sessions that made any attempt. Because this is only counting sessions attempting WebGL, which overall only a minority of sessions do as of 2014, this is biased towards users with recent machines and software, whence the relatively high rates.
  • Accelerated Layers counts the percentage of users running with OpenGL or Direct3D Layers. On all OSes except desktop Linux, this is always attempted by all sessions, aside from early startup crashes which we don't count. These curve therefore represent almost all of the user base. On Windows, we may be using either Direct3D 9 or, on Windows Vista and newer, Direct3D 10 or Direct3D 11 as represented by the corresponding curves. Direct3D 9 does not allow for fast (readback-free) compositing of WebGL canvases, so the Direct3D 10 and Direct3D 11 curves are important to understand the proportion of users who can enjoy heavy WebGL content in good conditions on Windows.

Legend

  • The vertical axis is the success rate, and the unit is percentage. That is, the ratio of people who actually get the feature over people who try to get it.
  • The horizontal axis is time.

Methodology

Crash reports contain information about which graphics features have been attempted, and which ones successfully initialized. This is true even if the crash itself has nothing to do with graphics features.

By analyzing these crash reports, we determine, among all crash reports sent on a given day, the number of crash reports reporting that a graphics feature had been attempted, and the number of crash reports where it had successfully initialized. The ratio between these two values is what is shown below as "success rate" percentage.

Obviously, getting this data from crash reports biases the results to a certain extent. Graphics features might be correlated with crashiness, for example, although crash-stats suggests that's not too marked.
